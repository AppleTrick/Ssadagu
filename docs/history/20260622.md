# 2026-06-22 작업 내역

## 1. AI 메타데이터/검색 LLM 제공자 교체: GMS → Groq

### 배경
- 기존 GMS(SSAFY OpenAI 프록시) API 키가 비활성화된 상태(`401 This GMS key is not Active`)인 것을 직접 호출 테스트로 확인
- `AIMetadataService`의 폴백 로직(빈 메타데이터/빈 필터 반환) 덕분에 서비스는 죽지 않았지만, 실제로는 AI 검색이 동작하지 않고 있었음

### 조치
- `AIMetadataService.java`: `gms.api.*` 설정 → `groq.api.*`로 교체 (`callGroqApi`, `GroqRequest`)
- `.env` / `application.properties`: `GMS_API_KEY` → `GROQ_API_KEY`, `GROQ_API_URL`, `GROQ_MODEL_PRIMARY=groq/compound`
- 직접 curl로 Groq API 요청 형식(`developer` role, `max_completion_tokens`) 호환 확인 후 적용

### 발견한 문제: `groq/compound`는 비전(이미지) 입력 미지원
- 이미지 포함 요청 시 `400 messages[1].content must be a string` 에러
- 해결: 이미지가 있으면 `meta-llama/llama-4-scout-17b-16e-instruct`(비전 지원 모델)로 자동 전환하도록 `callGroqApi` 내부에 분기 추가 (`groq.model.vision`)

## 2. Groq Rate Limit 문제 → Claude(Haiku 4.5) 병행 지원

### 배경
- 상품 16건을 동시 등록하자 `groq/compound`의 on-demand 티어 TPM 한도(8000)를 즉시 초과 (`429 rate_limit_exceeded`)
- `compound`는 내부적으로 여러 서브모델을 호출하는 에이전틱 모델이라 단순 텍스트 추출에도 토큰을 수천 개씩 소비

### 조치
- `ai.provider` 설정값(`groq` | `claude`)으로 LLM 제공자를 토글할 수 있게 구조 변경
  - `callAiApi()` 디스패처 추가 → `callGroqApi()` / `callClaudeApi()` 분기
  - `extractJsonFromResponse()`도 두 응답 형식(OpenAI 호환 `choices[]` vs Anthropic `content[]`)을 모두 파싱하도록 분기
  - `ClaudeRequest` 정적 클래스 추가 (Anthropic Messages API 형식, 이미지는 `source.type=url`)
- `.env`: `AI_PROVIDER=claude`, `ANTHROPIC_API_KEY`, `ANTHROPIC_MODEL=claude-haiku-4-5-20251001` 추가
- 현재 `ai.provider=claude`로 운영 중 (Groq 키는 유지, 언제든 되돌릴 수 있음)

## 4. 실제 버그 3건 발견 및 수정

### 4-1. 색상 정규화 불일치
- 등록 단계: LLM이 `canonicalColors`(13색 화이트리스트)만 저장 → "미드나이트" 같은 실제 표현이 손실되고 "검정"으로만 남음
- 검색 단계: LLM이 검색어의 색상을 화이트리스트 없이 원문 그대로 추출 ("미드나이트")
- → 등록된 "검정"과 검색의 "미드나이트"가 불일치해 매칭 실패
- **수정**: `extractMetadataFields()`에서 `colors`(실제 표현) + `canonicalColors`(화이트리스트)를 합쳐서 저장하도록 변경

### 4-2. 카테고리 코드 불일치
- 프론트엔드(`ItemRegistrationForm.tsx`)가 실제로 보내는 카테고리 코드: `ELEC, CLOT, BOOK, FURN, SPOR, BEAU, FOOD, KID, HOBB, ETC`
- `AIMetadataService.buildExtraHint()`는 존재하지 않는 코드(`DIGITAL, FASHION, FURNITURE`)를 기준으로 분기 → 항상 default 힌트만 적용되고 있었음
- 검색 단계에서는 LLM이 category를 자유 텍스트("전자기기")로 추출 → `category_code = ?` exact match 조건에 걸려 brand/product가 맞아도 검색 결과 0건
- **수정**: `buildExtraHint()`를 실제 코드 기준으로 교체, 검색 단계 `category`에 `condition`/`sort`/`tradeType`과 동일한 화이트리스트 검증 추가

### 4-3. 상품 이미지 URL이 가짜 주소였음
- `S3Service.uploadImage()`가 `https://{bucket}.s3.{region}.amazonaws.com/...` 형식(실제 AWS S3 URL)을 반환하고 있었으나, 실제 저장소는 MinIO → **존재하지 않는 URL**
- **수정**: `s3.public.base-url`(`https://s3.changhee.dev`) 설정을 추가해 실제 MinIO 공개 경로(`{base}/{bucket}/{fileName}`)를 반환하도록 변경

## 5. 인프라 변경: MinIO 공개 접근 허용

- 기존 `s3.changhee.dev` 라우팅에 `tailscale-only` 미들웨어가 걸려 있어 Claude/Groq 같은 외부 LLM 서버가 이미지 URL을 가져올 수 없었음
- 사용자 확인 후 `docker-compose.yml`의 `minio-api` 라우터에서 `tailscale-only` 미들웨어 제거
- MinIO 버킷(`ssadagu-image`) 정책을 `mc anonymous set download`로 공개 읽기 설정
- 결과: 외부에서 `https://s3.changhee.dev/ssadagu-image/{file}` 정상 접근(200) 확인, Claude가 실제로 이미지를 분석하는 것까지 확인

**⚠️ 보안 영향**: 이제 누구나 업로드된 상품 이미지 URL을 직접 열람할 수 있음 (관리자 콘솔 `minio.{TAILSCALE_DOMAIN}`은 별도 라우터로 그대로 비공개 유지됨).

## 6. 검증 측정

- **복합조건 검색** ("100만원 이하 검정 맥북 프로 미개봉", 정답 1건 + 오답 5건 설계): 기존 키워드 검색 0/1, AI 검색 1/1 정답 + 오답 5/5 정확히 제외 (정밀도·재현율 100%)
- **표현 변형 매칭** (18건): 기존 키워드 검색 0/18 → AI 검색 5/18(버그 수정 전) → **15/18(83%, 버그 수정 후)**
- 등록 16건 동시 처리 시 Groq rate limit으로 14/16건이 즉시 폴백(빈 메타데이터) 처리된 것을 로그로 확인 → "API 실패 시 안전 폴백" 설계가 실제로 다수 작동
