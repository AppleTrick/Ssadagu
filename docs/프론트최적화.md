#### 1. `staleTime: 0` 으로 인한 캐시 비활성화 문제

| 항목 | 내용 |
|------|------|
| **파일** | `apps/web/src/views/my-sales/ui/MySalesPage.tsx` |
| | `apps/web/src/views/chat-list/ui/ChatListPage.tsx` |
| **문제** | `staleTime: 0, gcTime: 0` 설정으로 인해 페이지 이동 시마다 TanStack Query 캐시가 즉시 무효화됨. 매 진입마다 네트워크 요청을 다시 보내고, 데이터 수신 전까지 **빈 화면(스켈레톤 또는 공백)이 노출**되어 깜빡임의 주 원인이었음. |
| **해결** | `staleTime: 30_000` (30초), `gcTime: 300_000` (5분)으로 변경. 30초 이내 재방문 시 캐시된 데이터를 즉시 표시하여 깜빡임 제거. |
| **효과** | 판매내역·채팅목록 페이지 전환 시 빈 화면 깜빡임 제거 |

#### 2. `ItemCard` 컴포넌트 메모이제이션 누락

| 항목 | 내용 |
|------|------|
| **파일** | `apps/web/src/entities/product/ui/ItemCard.tsx` |
| | `apps/web/src/widgets/product-list/ui/ProductList.tsx` |
| **문제** | `ItemCard` 가 `React.memo` 없이 export 되어, 상위 상태(예: 검색어, 찜 여부)가 변할 때마다 **목록 전체 카드가 불필요하게 리렌더링**됨. 또한 `onClick={() => ...}`, `onWishClick={(e) => ...}` 를 인라인으로 작성해 매 렌더마다 새 함수가 생성되어 memo의 props 비교를 무력화시킴. |
| **해결** | `ItemCard` 를 `memo(ItemCard)` 로 감쌈. `ProductList` 에 `MemoizedItem` 래퍼 컴포넌트를 추가해 `onNavigate`, `onLikeClick` prop을 안정적 참조로 전달. `handleLikeClick` 을 `useCallback` 으로 감쌈. `console.log('Current Token:', accessToken)` 디버그 로그 제거. |
| **효과** | 찜 클릭·스크롤·검색 시 변경되지 않은 카드는 리렌더링 생략 |

#### 3. 채팅방 스크롤 Layout Thrashing

| 항목 | 내용 |
|------|------|
| **파일** | `apps/web/src/views/chat-room/ui/ChatRoomPage.tsx` |
| **문제** | 이전 메시지 페이지 로드 후 스크롤 위치 복원 코드에서 `messagesAreaRef.current.scrollHeight` (레이아웃 읽기) → `messagesAreaRef.current.scrollTop = ...` (레이아웃 쓰기)를 **동기로 연속 실행**. 브라우저가 강제로 레이아웃을 재계산(Forced Synchronous Layout)하여 프레임이 버벅거림. |
| **해결** | `requestAnimationFrame` 콜백 안에서 scrollTop 쓰기를 수행. `prevScrollHeightRef` 값을 먼저 로컬 변수에 저장 후 0으로 초기화하여 동기 flush 이슈 방지. |
| **효과** | 채팅방 위로 스크롤해 이전 메시지 로드 시 화면 흔들림 제거 |

---

#### 4. 채팅 입력창 높이 자동조절 Layout Thrashing

| 항목 | 내용 |
|------|------|
| **파일** | `apps/web/src/widgets/chat-input/ui/ChatInputArea.tsx` |
| **문제** | 키 입력마다 `el.style.height = 'auto'` (쓰기) → `el.scrollHeight` 읽기 → `el.style.height = ... + 'px'` (쓰기) 순서로 실행되어 **매 키스트로크마다 브라우저 레이아웃 재계산** 발생. 특히 한국어 IME 입력 시 더 빈번하게 트리거되어 타이핑 중 프레임 드랍. |
| **해결** | `resizeTextarea` 함수 내부를 `requestAnimationFrame` 콜백으로 감쌈. 레이아웃 읽기/쓰기가 현재 프레임의 렌더링 이후 배치 처리됨. |
| **효과** | 채팅 입력 시 프레임 드랍 및 타이핑 지연 제거 |

#### 5. 이미지에 `width`/`height` 속성 없음 (CLS)

| 항목 | 내용 |
|------|------|
| **파일** | `apps/web/src/entities/product/ui/ItemCard.tsx` |
| | `apps/web/src/shared/ui/ImageCarousel.tsx` |
| **문제** | `<img>` 태그에 `width`, `height` 속성이 없어 브라우저가 이미지 로드 전까지 공간을 확보하지 못함. 이미지 로드 완료 시 **레이아웃이 갑자기 밀리는 CLS(Cumulative Layout Shift)** 발생. 상품 목록 스크롤 중 카드들이 갑자기 이동하는 현상의 원인. |
| **해결** | `ItemCard` 의 썸네일 이미지에 `width={100} height={100}` 추가 (CSS는 기존 100×100 유지). 캐러셀 첫 번째 이미지는 `loading="eager"`, 나머지는 `loading="lazy"` 설정. |
| **효과** | 상품 목록 스크롤 시 레이아웃 쉬프트 제거, 대역폭 절약 |

| **효과** | 앱 진입 시 흰 화면 깜빡임 제거 |

#### 7. 검색창 열기 `setTimeout(50ms)` 딜레이

| 항목 | 내용 |
|------|------|
| **파일** | `apps/web/src/widgets/header/ui/HeaderMain.tsx` |
| **문제** | `setSearchOpen(true)` 후 `setTimeout(() => inputRef.current?.focus(), 50)` 으로 50ms를 기다려 포커스를 줌. 검색 아이콘 클릭 시 **50ms의 눈에 띄는 반응 지연** 발생. |
| **해결** | `setTimeout` 제거 후 `useLayoutEffect` 로 교체. `searchOpen` 상태가 true로 바뀌어 DOM에 input이 나타나는 즉시 (`useLayoutEffect` 는 페인트 전에 실행) 포커스를 줌. |
| **효과** | 검색창 열릴 때 즉각적인 포커스, 딜레이 없음 |

#### 8. Pull-to-Refresh 조건부 마운트/언마운트

| 항목 | 내용 |
|------|------|
| **파일** | `apps/web/src/views/home/ui/HomePage.tsx` |
| **문제** | `pullY > 4` 조건에 따라 `PullIndicator` 가 매 픽셀 이동마다 마운트/언마운트 반복. React가 DOM 노드를 생성/제거할 때마다 **불필요한 페인트 비용** 발생. |

| **효과** | 홈 스크롤 중 Pull 인디케이터 깜빡임 제거 |

#### 9. 폰트 `font-display` 전략 없음 (FOIT)

| 항목 | 내용 |
|------|------|
| **파일** | `apps/web/src/shared/styles/global.ts` |
| **문제** | CDN에서 Pretendard 폰트를 로드할 때 `font-display` 지시어가 없음. 기본값인 `auto` (대부분의 브라우저에서 `block` 처럼 동작)로 인해 폰트 로드 전까지 **텍스트가 보이지 않는 FOIT(Flash of Invisible Text)** 현상 발생. 느린 네트워크에서 특히 두드러짐. |
| **해결** | `@font-face { font-display: swap; }` 추가. 폰트 로드 전 시스템 폰트를 먼저 표시하고 로드 완료 후 교체하여 텍스트가 항상 즉시 보임. |
| **효과** | 앱 초기 진입 시 텍스트 깜빡임 제거 |

#### 11. 이미지 캐러셀 `key={idx}` (배열 인덱스를 key로 사용)

| 항목 | 내용 |
|------|------|
| **파일** | `apps/web/src/shared/ui/ImageCarousel.tsx` |
| **문제** | `images.map((src, idx) => <ImageWrapper key={src + idx} ...>` 처럼 `src + idx` 를 key로 사용. Dot 컴포넌트는 `key={idx}` 만 사용. 이미지 배열이 변경될 때 React가 올바른 diffing을 하지 못해 **기존 DOM 노드를 재사용하지 못하고 불필요하게 재생성**하는 경우 발생. |
| **해결** | `ImageWrapper` 의 key를 `key={src}` 로 변경 (URL이 고유 식별자). `Dot` 의 key도 `key={src}` 로 변경하여 안정적인 식별자 사용. |
| **효과** | 이미지 목록 변경 시 React 재조정(reconciliation) 정확도 향상 |

## WebView 네이티브 체감 최적화

**배경:** 이 앱은 Next.js 웹앱을 WebView에 올린 구조로, 네이티브 앱 대비 체감 속도 차이가 발생함. 브라우저 렌더링 엔진의 구조적 한계 안에서 최대한 네이티브에 근접하게 최적화.

---

#### 12. WebView 300ms 탭 딜레이 제거

| 항목 | 내용 |
|------|------|
| **파일** | `apps/web/src/shared/styles/global.ts` |
| **문제** | WebView(모바일 브라우저)는 기본적으로 탭(클릭) 이벤트 발생 전 **300ms를 대기**함. 더블탭 줌 동작인지 확인하기 위한 브라우저 기본 동작. 모든 버튼·링크가 누른 즉시 반응하지 않고 미세하게 느리게 느껴지는 원인. |
| **해결** | `touch-action: manipulation` 을 `html`, `body`, `button`, `a`, `input`, `textarea` 전역 적용. 더블탭 줌을 비활성화하여 브라우저가 탭을 기다리지 않고 즉시 이벤트 발생. |
| **효과** | 모든 버튼·링크 탭 반응 즉각화, 300ms 딜레이 완전 제거 |

#### 14. iOS WebView 네이티브 관성 스크롤 활성화

| 항목 | 내용 |
|------|------|
| **파일** | `apps/web/src/shared/styles/global.ts` 및 전체 스크롤 컨테이너 |
| **문제** | iOS WebView의 기본 스크롤은 JS 스레드를 경유하여 처리되어 네이티브 스크롤보다 느리고 관성(momentum)이 부족함. 손가락을 떼면 즉시 멈추거나 뚝뚝 끊기는 느낌. |

| **적용 파일** | `global.ts`, `HomePage`, `ChatListPage`, `ChatRoomPage`, `MySalesPage`, `MyWishlistPage`, `MyPurchasesPage`, `MyPage`, `ProductDetailPage` |

#### 15. Fixed 요소 GPU 합성 레이어 분리

| 항목 | 내용 |
|------|------|
| **파일** | `apps/web/src/widgets/header/ui/HeaderMain.tsx` |
| | `apps/web/src/widgets/header/ui/HeaderBack.tsx` |
| | `apps/web/src/widgets/bottom-nav/ui/BottomNav.tsx` |
| **문제** | `position: fixed` 인 헤더와 하단 탭바가 스크롤할 때마다 CPU가 매 프레임 위치를 재계산하고 repaint함. 스크롤 중 헤더/탭바가 미세하게 흔들리거나 프레임이 떨어지는 원인. |
| **해결** | `transform: translateZ(0)` + `will-change: transform` 적용. 브라우저가 해당 요소를 **독립된 GPU 합성 레이어**로 분리하여 스크롤 중 CPU가 전혀 건드리지 않음. |
| **효과** | 스크롤 중 헤더·탭바 repaint 완전 차단, 60fps 유지 |

#### 16. 스크롤 컨테이너 GPU 레이어 예약

| 항목 | 내용 |
|------|------|
| **파일** | 전체 `ContentArea`, `MessagesArea` 스타일 컴포넌트 |
| **문제** | 스크롤이 시작되는 순간 브라우저가 GPU 레이어를 생성하면서 **첫 번째 프레임이 버벅**거림. 특히 목록이 많을수록 첫 스크롤 시작점에서 끊기는 느낌 발생. |
| **해결** | `will-change: scroll-position` 적용으로 브라우저가 스크롤 전부터 GPU 레이어를 미리 준비. 스크롤 시작 첫 프레임부터 부드럽게 동작. |
| **효과** | 스크롤 시작점 버벅거림 제거, 첫 터치부터 부드러운 스크롤 |
